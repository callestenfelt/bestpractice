## 0. Tl;dr

Three things changed in this session. The taxonomy grew; the filter UI
moved out of a modal into a right-hand drawer mirroring the existing left
sidebar; and the whole sidebar now uses Phosphor Icons so users can scan
page-types and components by glyph.

1. **Taxonomy** — 4 new page types, 17 new components, 1 new phase. Now
   **21 page types / 63 components / 11 phases**. See `uploads/taxonomy-additions.md`
   for the canonical list. Sidebar counts in v3 already reflect this.
2. **Filters drawer** — the `<dialog>` modal and the in-page "Filters"
   button are gone. Filters now live in a sticky right-hand drawer
   (`.filters-rail`) toggled from a settings icon in the topbar with the
   shortcut **`Ctrl/Cmd + ,`** (mirrors the existing `Ctrl/Cmd + .` for
   the left sidebar).
3. **Phosphor Icons** — replaces inline SVGs in the sidebar nav and in
   the topbar chrome (sidebar toggle, search field icon, filters toggle).
   Three weights (Regular / Bold / Duotone) are loaded and a small
   selector at the bottom of the sidebar lets the user A/B them; final
   weight choice is the user's call.

## 2. Filters drawer

### 2.1 What replaced the modal

The previous "Filters" button in the page heading, the floating
`.filters-btn__count` badge, the entire `<dialog class="filters-dialog">`
node, and its JS open/close handlers are all **deleted**. Don't carry
them forward — they were a deliberate scrap.

### 2.2 New structure

The drawer is a sibling of `.sidebar` inside `.layout`:

```html
<div class="layout">
  <aside class="sidebar" id="sidebar"> … </aside>
  <main class="content"> … </main>
  <aside class="filters-rail" id="filters-rail" aria-label="Filters">
    <div class="filters-rail__inner">
      <header class="filters-rail__header">
        <h2 class="filters-rail__title">Filters</h2>
        <span class="filters-rail__count" data-role="filters-count" hidden></span>
      </header>
      <div class="filters-rail__body">
        <!-- .rail__section with the phase checklist -->
        <!-- .rail__section with #toggle-sitewide -->
      </div>
    </div>
  </aside>
</div>
```

The layout grid is now `var(--sidebar-w) minmax(0, 1fr) var(--filters-w)`.
Each rail's width custom property collapses to `0px` when its
`data-…-open` attribute on `.app` is `"false"`, animating via
`transition: grid-template-columns 180ms`.

### 2.3 DOM/JS contract (add to BUILD_NOTES §3)

| Attribute / class | Purpose |
|---|---|
| `.app[data-filters-open="true"\|"false"]` | Drawer state. Persisted to `localStorage` under `bp:filters-open`. |
| `[data-action="toggle-filters"]` | Click target on the topbar settings button. |
| `[data-action="close-filters"]` | (No longer used — the dialog close button is gone. Drop from any inherited code.) |
| `#filters-rail` | The drawer root. `aria-controls` target. |
| `.filters-scrim` | Mobile overlay scrim; only visible <960px when drawer is open. |
| `.topbar__toggle--filters` | The settings button itself. Carries `aria-expanded` in sync with `data-filters-open`. |
| `.topbar__toggle-count` | Optional count badge inside the button (e.g. "3 off"). Hidden when no filters are off and site-wide is off. |
| `[data-role="filters-count"]` | Same content as the topbar badge, mirrored inside the drawer header so users see the state from either side. |

The filter checkboxes themselves keep their old contract:
`<input type="checkbox" name="phase" value="{slug}">` and
`<input type="checkbox" id="toggle-sitewide">`. Filter JS does **not**
care that they moved from a `<dialog>` to an `<aside>`.

### 2.4 Keyboard shortcut

| Keys | Action |
|---|---|
| `Ctrl/Cmd + .` | Toggle left sidebar (existing, unchanged) |
| `Ctrl/Cmd + ,` | Toggle right filters drawer (new) |

Chosen because comma and period are adjacent on the keyboard and pair
naturally with the left/right rails. The handler lives in
`prototype/js/sidebar.js` and ignores the event when shift/alt are also
held, so it won't shadow OS-level shortcuts. Document this in the user-
facing keyboard help if one exists; otherwise the `title` attributes on
the two buttons surface them on hover.

### 2.5 Settings icon

The topbar settings button uses **Phosphor `sliders-horizontal`** (see §3
below). Its label is "Toggle filters" and its `title` includes the
shortcut. Don't restyle it — `.topbar__toggle--filters` already handles
the slightly wider hit target needed to accommodate the optional count
badge.

### 2.6 Responsive behaviour

Under **960px** both rails fix-position over the content with their own
scrim. The breakpoint is shared with the left sidebar — change it in one
place if you change it at all. `prefers-reduced-motion` disables the
grid-template-columns transition globally; the rails snap rather than
slide.

## 3. Phosphor Icons

### 3.1 The package

- **Library:** [Phosphor Icons](https://phosphoricons.com/)
- **CDN used in the prototype:** `https://unpkg.com/@phosphor-icons/web@2.1.1/`
- **Weights loaded:** `regular`, `bold`, `duotone` (three separate
  stylesheets in `<head>`).
- **License:** MIT.

### 3.2 Production install

Vendor it, don't keep the CDN. Two reasonable approaches:

1. **npm + self-host the woff2.** `npm i @phosphor-icons/web`, then ship
   the chosen weight's `style.css` + `Phosphor.woff2` (and friends if
   you keep multiple weights) under `static/icons/phosphor/`. Adjust the
   `@font-face` `src` to match.
2. **Stick with web font, vendored.** Copy the `src/regular/` (and
   optionally `bold/` and `duotone/`) folders into
   `static/icons/phosphor/` and link the local `style.css`.

Whichever you pick, drop the three unpkg `<link>` tags from `<head>` and
replace with local ones. The icon names and class structure don't
change.

### 3.3 Which weight ships?

The prototype loads all three so the user can compare. Until the user
commits, the default is **Regular** (the `ph` class prefix) and the
selector at the bottom of the sidebar lets them flip in-browser. The
choice is persisted to `localStorage` under `bp:icon-weight`.

When the user picks one (likely Regular), do three things:

1. Remove the other two `<link>` tags from `<head>`.
2. Remove the `.icon-weight` selector markup (the three buttons above the
   sidebar footer) and the inline `<script>` that drives it.
3. Remove the `localStorage` read of `bp:icon-weight` — every icon ships
   with the chosen weight class baked in (`ph` for Regular).

Until then, treat the weight switcher as a transient debug control. It
is **not** intended to ship to end users.

### 3.4 Markup pattern

Icons are `<i>` elements with two classes — a weight prefix and an icon
name:

```html
<i class="ph ph-house" aria-hidden="true"></i>       <!-- Regular -->
<i class="ph-bold ph-house" aria-hidden="true"></i>  <!-- Bold -->
<i class="ph-duotone ph-house" aria-hidden="true"></i> <!-- Duotone -->
```

All Phosphor icons in the prototype carry `aria-hidden="true"`. They're
decorative — the adjacent link text is the accessible label. If you ever
use one as the *only* content of an interactive element (e.g. an icon-
only button), drop the `aria-hidden` and add a sibling `<span class="sr-only">`
or use `aria-label` on the parent. The three topbar chrome buttons all
have `aria-label` already.

### 3.5 Where icons are used in v3

| Surface | Icon |
|---|---|
| Topbar — sidebar toggle | `ph-sidebar-simple` |
| Topbar — search field | `ph-magnifying-glass` (keeps the `.search__icon` class for positioning) |
| Topbar — filters toggle | `ph-sliders-horizontal` |
| Sidebar — every page-type link | one icon per slug, see §3.7 |
| Sidebar — every component link | one icon per slug, see §3.7 |

The inline chevrons inside `<details class="consideration">` and
`<details class="sub">`, the close-X glyphs in the (removed) dialog, and
the checkbox tick inside `.checklist__item input:checked::before` are
**still inline SVGs**. They're geometric, sized to their container, and
not worth converting. Leave them.

### 3.6 CSS hooks

Inline `<style>` block in `page-type-v3.html` defines:

- `.sidebar__link i.ph` (and the bold/duotone variants) — 16px, gray-10
  by default, gray-12 on hover, **blue-11 on `[aria-current="page"]`**.
  The active-link icon picking up the accent is intentional — it
  reinforces the current-page signal beyond background alone.
- `.topbar__toggle i.ph` — 18px (slightly larger than sidebar so the
  chrome reads at a glance).
- `.search__icon.ph` — 14px, positioning inherited from the existing
  `.search__icon` rules in `components.css`.

When you move v3 into Flask, lift these rules into the appropriate
stylesheet — probably `sidebar.css` (icon rules for the rail) and
`components.css` (search and topbar overrides). Don't leave them as an
inline `<style>` block.

## 6. Open questions

1. **Phosphor weight.** Confirm with the user. Once chosen, strip the
   other two stylesheets and the weight switcher per §3.3.
2. **Self-host vs CDN for Phosphor.** Default: self-host. Confirm before
   wiring.
3. **Icon column in the seed.** Mapping in §3.7 is design's
   recommendation; the user may want to override individual slugs
   editorially before the seed lands.
4. **Filters drawer default state.** Currently defaults to **open** on
   first visit. The same convention as the left sidebar. If the user
   prefers it closed by default, change the `data-filters-open="true"`
   to `"false"` in `page-type-v3.html` and adjust the `localStorage`
   default in `sidebar.js`.
5. **`Ctrl/Cmd + ,` collision.** On macOS, `Cmd + ,` is the system
   "Preferences" shortcut inside apps. The browser doesn't claim it, so
   it's free for web pages — but if a user has a browser extension that
   does, the shortcut may not reach the page. Acceptable risk; flag if
   the user reports it.
